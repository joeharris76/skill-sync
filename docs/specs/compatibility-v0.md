# skill-sync Compatibility & Portability Specification v0

This document defines how skill-sync handles agent compatibility, portable
materialization, and project-specific configuration injection.

## Agent Targets

skill-sync supports materializing skills into multiple agent-specific directories
from a single project manifest.

### Supported Targets

| Target | Label | Default Directory | Reads Frontmatter | AGENTS.md |
|--------|-------|-------------------|-------------------|-----------|
| `claude` | Claude Code | `.claude/skills` | Yes | No |
| `codex` | OpenAI Codex | `.codex/skills` | Yes | Yes |
| `generic-mcp` | Generic MCP | `.agent/skills` | No | No |

### Compatibility Checking

When a skill is materialized to a target, skill-sync checks for features the
target cannot support:

| Feature | Claude | Codex | Generic MCP |
|---------|--------|-------|-------------|
| SKILL.md frontmatter | Full | Full | Not read directly |
| `allowed-tools` | Supported | Ignored | Ignored |
| `scripts/` | Supported | Supported | Resources only |
| `references/` | Supported | Supported | Supported |
| `AGENTS.md` discovery | No | Yes | No |

### Fallback Behavior

When a skill uses features unsupported by a target:

- **Warning** (default): Skill is materialized with a diagnostic warning.
  The unsupported feature is present in the files but will be ignored at
  runtime. Example: `allowed-tools` is kept in SKILL.md but Codex ignores it.

- **Error**: Only when the skill explicitly declares itself incompatible
  via `targets: { codex: false }` in skill.yaml. Materialization
  is blocked.

skill-sync never silently drops content. Unsupported features remain in the
files — the target agent simply doesn't use them. This means:

1. Skills are never mutated or stripped during materialization.
2. The same files go to all targets.
3. Compatibility diagnostics inform the user, not gatekeep the install.

## Portability

### The Problem

Skills authored on a developer's machine often contain references to:
- `~/.claude/skills/` (home directory skill store)
- `/Users/joe/...` (absolute user paths)
- Symlinks that resolve to machine-local locations

These break in CI, web-based Claude, remote containers, or another
developer's machine.

### Portable vs Non-Portable Install Modes

| Mode | Portable | Use Case |
|------|----------|----------|
| `mirror` | Yes | Default. Full copy + integrity tracking. |
| `copy` | Yes | Lightweight copy without lock tracking. |
| `symlink` | No | Local development only. |

### Portability Validation

`skill-sync validate` and `skill-sync doctor` scan installed skills for
non-portable patterns:

- `~/` home directory references in SKILL.md body or references
- Absolute paths (`/Users/`, `/home/`, `C:\Users\`)
- References to `.claude/skills/` or `.codex/skills/` global stores

Each match produces a `non-portable-path` diagnostic with the file, line
number, and offending pattern.

### Recommendation for CI

```yaml
# skill-sync.yaml for CI-safe portable installs
install_mode: mirror   # Never symlink
targets:
  claude: .claude/skills
```

Verify with: `skill-sync validate --exit-code`

## Config Injection

### The Problem

Generic shared skills need project-specific values (test runner, lint
command, source directories) but forking the entire skill to change one
value creates divergence.

### Solution: Layered Config Generation

skill-sync generates `skill-sync.config.yaml` during sync, combining:

1. **Skill defaults** from `skill.yaml` → `config_inputs[].default`
2. **Project overrides** from `skill-sync.yaml` → `config` section

Project values always win over skill defaults.

### Example Flow

Skill declares config inputs:
```yaml
# test/skill.yaml
config_inputs:
  - key: test.runner
    type: string
    description: "Test runner command"
    default: "pytest"
  - key: test.test_dir
    type: string
    description: "Test directory"
    default: "tests/"
```

Project overrides one value:
```yaml
# skill-sync.yaml
config:
  test:
    runner: "uv run pytest"
```

Generated output:
```yaml
# .claude/skills/skill-sync.config.yaml
# Generated by skill-sync. Do not edit manually.
test:
  runner: uv run pytest    # From project manifest (override)
  test_dir: tests/         # From skill default (preserved)
```

### Config Validation

`skill-sync validate` checks:
- Config keys match declared `config_inputs` (warns on unknown keys)
- Config references skills that are actually installed
- Types match declared `config_inputs` types (future)

### What Config Injection Does NOT Do

- It does not modify SKILL.md or any skill file content
- It does not template or substitute variables within skill instructions
- It does not conditionally include or exclude parts of a skill
- Skills read `skill-sync.config.yaml` at runtime — skill-sync only generates it

## Override Layering

### Per-Skill Install Mode Overrides

```yaml
# skill-sync.yaml
install_mode: mirror          # Project default
overrides:
  code:
    install_mode: symlink     # Active development on this skill
  todo:
    install_mode: copy        # Lightweight, no tracking needed
```

### Future: Content Overrides (v0.2+)

Content-level overrides (e.g., replacing a specific reference file or
appending instructions to a SKILL.md) are not supported in v0. If a
project needs to modify a skill's content, the current approach is:

1. Fork the skill locally (copy, modify, use as a local source)
2. Promote changes back upstream when ready

v0.2 may introduce structured overlay files that inject or replace
specific sections without forking.
